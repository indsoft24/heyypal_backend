## Expert Onboarding & Admin Approval - Backend Requirements

This document specifies how the backend should support expert onboarding for the HeyyPal apps. It is written so another backend engineer or AI agent can continue implementation consistently.

### 1. Roles and High-Level Behaviour

- Users can choose to use the app as:
  - **Normal user** (role: `user`)
  - **Expert** (role: `expert`)
- For **normal users**:
  - Collect minimal profile data: `name`, `gender` (optional), `dateOfBirth` (optional), `phone`.
  - Mark `role = user`, `profileCompleted = true`.
  - They can explore and use the app immediately.
- For **experts**:
  - `role = expert`.
  - They must submit an **expert application** which is **reviewed by admin**.
  - The expert’s profile is only activated as an expert when an admin **approves** the application.
  - Until approval:
    - The user is allowed to use the app normally, but **as a normal user only**.
    - In the user object we keep `expertStatus = pending` and (optionally) `expertType`.

### 2. User Model Extensions

Extend the `User` entity (`users` table) with:

- `gender: string | null`
- `dateOfBirth: string | null` (ISO `YYYY-MM-DD` as string is fine)
- `expertType: 'supportive' | 'professional' | null` (enum `ExpertType`)

Existing fields:

- `role: 'user' | 'expert' | 'admin'`
- `expertStatus: 'pending' | 'approved' | 'rejected' | null`
- `profileCompleted: boolean`

Rules:

- When completing profile:
  - If `role = user`:
    - `expertStatus = null`
    - `expertType = null`
  - If `role = expert`:
    - `expertStatus = pending` (the actual expert profile still needs admin approval)
    - `expertType` is **not** decided at this step (decided in expert application).

### 3. Expert Profile Domain

Create a separate module `experts` with:

- Entity: `ExpertProfile` (`expert_profiles` table)
- Service: `ExpertsService`
- Controller: `ExpertsController`
- Module: `ExpertsModule` (imported in `AppModule`)

#### 3.1 ExpertProfile Entity

Fields:

- `id: number`
- `user: User` (`ManyToOne`, required)
- `type: 'supportive' | 'professional'` (enum `ExpertType`)
- `category: one of`:
  - `Fitness`
  - `Emotional help`
  - `Relationships`
  - `Skincare`
  - `Comfort Zone`
  - `Fashion`
  - `Comedian`
  - `Story Telling`
- `bio: string` (max 300 chars)
- `languagesSpoken: string[]` (simple-array; options: English, Hindi, Tamil, Telugu, Malayalam, Kannada)
- `photos: string[] | null` (URLs/paths for professional photos; at least 2 uploaded)
- `introVideoUrl: string | null` (original uploaded video URL)
- `introVideoCompressedUrl: string | null` (compressed video URL)
- `degreeCertificateUrl: string | null` (required for professional experts)
- `aadharUrl: string | null` (required for professional experts)
- `createdAt`, `updatedAt` timestamps

Notes:

- **Storage format** for media is URL/string paths; actual file storage/compression pipelines can be implemented separately (e.g. S3, Cloud Storage, local).
- The mobile/web clients are expected to upload the files first, obtain URLs, and then call the expert profile API with those URLs.

### 4. Expert Application Flow (API Design)

#### 4.1 Complete profile (common to all users)

Existing endpoint:

- `POST /api/auth/profile/complete`
  - Guard: `JwtAuthGuard`
  - Body:
    - `name: string`
    - `phone: string`
    - `role: 'user' | 'expert'`
    - (optionally extend later with `gender`, `dateOfBirth` if needed on this endpoint)
  - Behaviour:
    - Update user’s basic profile fields.
    - Encrypt phone into `phoneEnc`.
    - Set `role`.
    - If `role = 'expert'`, set `expertStatus = 'pending'` and leave ability to use app as normal user but mark them as potential expert.
    - Mark `profileCompleted = true`.

#### 4.2 Submit Expert Profile

New endpoint:

- `POST /api/experts/profile`
  - Tags: `experts`
  - Guards: `JwtAuthGuard`
  - Auth: `@ApiBearerAuth()`
  - Request body (DTO):
    - `type: 'supportive' | 'professional'` (enum `ExpertType`)
    - `category: ExpertCategory` (see list above)
    - `bio: string` (required, max 300 chars)
    - `languagesSpoken: string[]` (non-empty; subset of [English, Hindi, Tamil, Telugu, Malayalam, Kannada])
    - `photos: string[]` (non-empty; at least 2 image URLs)
    - `introVideoUrl: string` (original high-quality video URL)
    - `introVideoCompressedUrl: string` (compressed video URL, generated by media pipeline)
    - `degreeCertificateUrl?: string` (required if `type = professional`)
    - `aadharUrl?: string` (required if `type = professional`)
  - Behaviour:
    - Load current user by `userId`.
    - Require `user.role === 'expert'`:
      - If not, `ForbiddenException("Only expert users can submit expert profile")`.
    - Upsert or create a new `ExpertProfile` linked to the user.
    - Set `user.expertStatus = 'pending'`.
    - Set `user.expertType = dto.type`.
    - Save both `User` and `ExpertProfile`.
    - Return the created/updated `ExpertProfile` with the related user info.

Validation logic:

- If `dto.type === 'professional'`:
  - `degreeCertificateUrl` and `aadharUrl` must be non-empty.
- Ensure `languagesSpoken` and `photos` are non-empty arrays.
- Enforce `bio.length <= 300`.

### 5. Admin Panel Behaviour

Use the existing `AdminModule` and extend it with expert-related functionality.

#### 5.1 List expert requests

Existing route signature:

- `GET /api/admin/experts`
  - Guards: `AuthGuard('jwt')`, `AdminRoleGuard`, `@AdminOnly()`
  - Return:
    - List of expert applications with **full context** for the admin UI:
      - User metadata:
        - `id`
        - `name`
        - `email`
        - `role`
        - `expertStatus`
        - `expertType`
        - `gender`
        - `dateOfBirth`
        - `createdAt`
      - Expert profile details:
        - `id`
        - `type`
        - `category`
        - `bio`
        - `languagesSpoken`
        - `photos` (list of URLs)
        - `introVideoUrl`
        - `introVideoCompressedUrl`
        - `degreeCertificateUrl`
        - `aadharUrl`
        - `createdAt`, `updatedAt`

Admin UI guidelines (data this endpoint must provide clearly):

- Make it easy to show a **card** or **detail page** with:
  - Basic user info on top.
  - Expert type (Supportive/Professional).
  - Category & languages as chips/badges.
  - Short bio text.
  - Photo gallery (thumbnails).
  - Playable intro video (using `introVideoCompressedUrl`).
  - For professional experts: distinct section for uploaded degree/certificate and Aadhaar document (downloadable links/thumbnails).
  - Expert status badge: Pending / Approved / Rejected.

#### 5.2 Approve / Reject expert

Existing routes:

- `POST /api/admin/experts/:id/approve`
- `POST /api/admin/experts/:id/reject`

Expected behaviour:

- `:id` here should refer to the **user id** (or clearly documented; recommended: user id).
- On **approve**:
  - Set `user.expertStatus = 'approved'`.
  - `user.role` should remain `expert`.
  - Optionally, notify the user (via push/email) that they are approved.
- On **reject**:
  - Set `user.expertStatus = 'rejected'`.
  - `user.role` can remain `expert`, but the client must treat `expertStatus !== 'approved'` as **not an active expert**.
    - Alternatively, if business logic requires, you can down-grade role back to `user`. In that case, ensure the mobile client continues to function correctly (this is a product decision).

### 6. Mobile Client Behaviour (for reference)

The Android app behaviour that the backend must support:

- On registration/login:
  - If `profileCompleted = false`, user is taken to **Complete Profile** screen.
  - On Complete Profile, user chooses:
    - **Normal user**:
      - Provide: `name`, `gender`, `dateOfBirth`, `phone`.
      - Call `POST /api/auth/profile/complete` with `role = 'user'`.
      - On success: user can fully explore the app.
    - **Expert**:
      - Provide at least `name`, `phone` (gender/DOB can be collected here or in later steps).
      - Call `POST /api/auth/profile/complete` with `role = 'expert'`.
      - Then navigate through expert onboarding steps:
        1. Choose `type`: Supportive or Professional expert.
        2. Fill form:
           - Mobile number (already captured/verified at auth-level).
           - Date of Birth.
           - Languages (multi-select).
           - Short bio (2–3 lines, max 300 characters, mention area of expertise).
           - Category (single-choice from predefined list).
        3. Upload media:
           - At least 2 professional photos.
           - 10–12 sec intro video (upload original, then call media service to compress).
           - If Professional:
             - Degree/certificate file.
             - Aadhaar image/PDF.
      - Once all data and file URLs are ready, the client calls:
        - `POST /api/experts/profile` with the payload described in **4.2**.
      - After submission:
        - The expert is considered **pending** (`expertStatus = pending`).
        - The client shows messaging like “Your expert profile is under review. Meanwhile, you can continue using HeyyPal as a normal user.”
        - Until `expertStatus = approved`, any expert-only features/UI must remain disabled.

### 7. Media & Compression (implementation hint)

Backend prompt for media handling:

- You may:
  - Accept file uploads via multipart endpoints and return URLs, or
  - Delegate upload to a separate media service (S3 pre-signed URLs, etc.), and only store returned URLs in expert profile.
- For video compression:
  - A background worker or media pipeline should:
    - Take `introVideoUrl` as input.
    - Generate a compressed version and store it.
    - Update `introVideoCompressedUrl` on the `ExpertProfile`.
  - Until compression is done, the client can fallback to `introVideoUrl`.

### 8. Security & Privacy

- Phone number (`phoneEnc`) must remain encrypted at rest.
- Aadhaar and degree/certificate files must be stored securely with appropriate access controls.
- Admin endpoints must:
  - Require a valid admin JWT (`AuthGuard('jwt')` + `AdminRoleGuard`).
  - Not be accessible to normal users or experts.

### 9. Acceptance Criteria

- Normal users:
  - Can register, complete profile, and use the app without touching expert flows.
- Experts:
  - Can select expert role and complete multi-step onboarding.
  - Data is persisted in `users` and `expert_profiles`.
  - `expertStatus` controls whether they are allowed to operate as experts.
- Admins:
  - Can view a complete picture of each expert application (user + expert profile + documents + media).
  - Can approve/reject experts with clear state transitions.
